1. خلاصه هدف سیستم
هدف سیستم، تبدیل فرآیند فعلی اکسل‌محور قیمت‌گذاری باسداکت / Busduct به یک وب‌اپ است که کاربر بتواند در آن پروژه یا پیشنهاد قیمت جدید تعریف کند، اسکوپ‌های پروژه را مشخص کند، برای هر اسکوپ چند لاین تعریف کند، قطعات هر لاین را با تعداد وارد کند، و سیستم به‌صورت خودکار بر اساس جدول مادر MTO/MTU و قیمت متریال پروژه، قیمت نهایی را محاسبه کند.

در فایل اکسل فعلی، جدول مادر شامل این منطق است:

Scope / Source + Part Name + Material + Consumption Value + Unit Price = Material Total Price

و سپس در شیت Line این منطق استفاده می‌شود:

Unit Price قطعه در یک اسکوپ = مجموع Total Price ردیف‌های MTO که Source و Name آن‌ها با قطعه انتخاب‌شده برابر است

و بعد:

Total Price لاین = Qty قطعه در لاین × Unit Price قطعه

بنابراین وب‌اپ باید در واقع همین اکسل را به یک سیستم دیتابیس‌محور، قابل کنترل، قابل توسعه، قابل گزارش‌گیری و قابل استفاده برای آفرهای واقعی تبدیل کند.

8.1 فرمول قیمت واحد پیش‌فرض
در ستون B، از B2 تا B91، فرمول زیر استفاده شده است:

=VLOOKUP(A:A,H:K,4,1)
هدف این فرمول این است که برای متریال موجود در ستون A، قیمت واحد را از جدول H:K پیدا کند.

اما این فرمول چند مشکل جدی دارد:

اول اینکه به‌جای A2 از کل ستون A:A استفاده شده است. در بعضی نسخه‌های Excel این ممکن است با implicit intersection کار کند، اما برای تبدیل به سیستم نرم‌افزاری نباید این رفتار مبهم مبنا قرار گیرد.

دوم اینکه پارامتر چهارم VLOOKUP برابر 1 است؛ یعنی جست‌وجوی تقریبی یا approximate match. برای داده‌های متنی متریال، این خطرناک است. اگر جدول مرتب نباشد یا متریال‌ها مشابه باشند، ممکن است قیمت اشتباه برگردد.

فرمول امن‌تر در اکسل باید شبیه این باشد:

=VLOOKUP(A2,$H:$K,4,FALSE)
یا بهتر از آن:

=XLOOKUP(A2,$H:$H,$K:$K)
اما در وب‌اپ اصلاً نباید با نام متریال جست‌وجو انجام شود؛ باید با material_id انجام شود.

25.1 اکسل نباید فقط کپی شود
فرمول‌های اکسل باید به منطق قابل تست در بک‌اند تبدیل شوند. یعنی سیستم نباید صرفاً یک فایل اکسل را در پشت صحنه پر کند. منطق باید در دیتابیس و کد محاسباتی باشد.

25.2 قیمت‌ها باید Snapshot شوند
اگر امروز یک آفر ساخته شد و فردا قیمت متریال تغییر کرد، آفر امروز نباید بدون اجازه تغییر کند. به همین دلیل جدول project_material_prices ضروری است.

25.3 محاسبه باید قابل ردیابی باشد
برای هر عدد نهایی، کاربر باید بتواند ببیند از کدام ردیف‌های MTO آمده است.

802,700,038.30 ریال

25.4 شناسه‌ها مهم‌تر از نام‌ها هستند
در اکسل، تطبیق بر اساس متن انجام شده است؛ مثلاً Source.Name و Name. اما در وب‌اپ باید تطبیق با ID انجام شود.

26. ریسک‌ها و ایرادهای فعلی داده
26.1 فرمول VLOOKUP خطرناک است
فرمول:

=VLOOKUP(A:A,H:K,4,1)
برای سیستم عملیاتی مناسب نیست. به‌خصوص چون جست‌وجوی تقریبی است و بعضی متریال‌ها چند بار تکرار شده‌اند.

26.2 بعضی Total Priceها قیمت واقعی نیستند
۳۰ ردیف وجود دارد که UNIT PRICE خالی است اما TOTAL PRICE مقدار غیرصفر دارد. این ردیف‌ها مربوط به ابعاد ناودانی و ابعاد نبشی هستند و محاسبه آن‌ها شبیه محاسبه وزن با ضریب 7.86 است.

پس نباید همه مقادیر TOTAL PRICE را کورکورانه به‌عنوان مبلغ ریالی در نظر گرفت.

26.3 ۲۰۰ ردیف Unit Price خالی دارند
این مورد باید در وب‌اپ هشدار داده شود. البته بسیاری از آن‌ها Total صفر دارند و شاید اطلاعاتی هستند، اما سیستم باید آن‌ها را طبقه‌بندی کند.

26.4 ۱۰ ردیف با فرمول اصلی تطابق ندارند
۱۰ ردیف مربوط به گلمپ های نگهدارنده وجود دارد که اگر با VALUE × UNIT PRICE محاسبه شوند، با مقدار TOTAL PRICE اکسل برابر نمی‌شوند. مقدار اکسل بیشتر شبیه استفاده از QTY × UNIT PRICE است.

بنابراین این آیتم باید با کارفرما بررسی شود و در سیستم با روش محاسبه مشخص ذخیره شود.

-- View: current offer line part with scope
CREATE OR REPLACE VIEW vw_offer_line_parts AS
SELECT
    o.id AS offer_id,
    o.offer_number,
    o.name AS offer_name,

    os.id AS offer_scope_id,
    s.id AS scope_id,
    s.code AS scope_code,
    s.source_name,

    ol.id AS offer_line_id,
    ol.line_name,

    olp.id AS offer_line_part_id,
    p.id AS part_id,
    p.name AS part_name,

    olp.qty,
    olp.unit_price,
    olp.total_price,
    olp.calculation_status

FROM offer_line_parts olp
JOIN offer_lines ol ON ol.id = olp.offer_line_id
JOIN offer_scopes os ON os.id = ol.offer_scope_id
JOIN offers o ON o.id = os.offer_id
JOIN scopes s ON s.id = os.scope_id
JOIN parts p ON p.id = olp.part_id;

-- View: MTO part unit price based on imported totals
-- This reproduces Excel SUMIFS at base/import level.
CREATE OR REPLACE VIEW vw_base_part_prices AS
SELECT
    mr.mto_version_id,
    mr.scope_id,
    s.code AS scope_code,
    s.source_name,
    mr.part_id,
    p.name AS part_name,
    SUM(COALESCE(mr.imported_total_price, 0)) AS base_part_unit_price,
    COUNT(*) AS mto_row_count
FROM mto_rows mr
JOIN scopes s ON s.id = mr.scope_id
JOIN parts p ON p.id = mr.part_id
WHERE mr.is_active = TRUE
  AND mr.is_price_relevant = TRUE
GROUP BY
    mr.mto_version_id,
    mr.scope_id,
    s.code,
    s.source_name,
    mr.part_id,
    p.name;

-- View: offer totals based on cached line part totals
CREATE OR REPLACE VIEW vw_offer_current_totals AS
SELECT
    o.id AS offer_id,
    o.offer_number,
    o.name AS offer_name,
    o.status,
    COALESCE(SUM(olp.total_price), 0) AS current_total_price
FROM offers o
LEFT JOIN offer_scopes os ON os.offer_id = o.id
LEFT JOIN offer_lines ol ON ol.offer_scope_id = os.id
LEFT JOIN offer_line_parts olp ON olp.offer_line_id = ol.id
GROUP BY
    o.id,
    o.offer_number,
    o.name,
    o.status;

-- This calculates one MTO row according to its calculation method.
-- =========================================================

CREATE OR REPLACE FUNCTION calculate_mto_row_total(
    p_calc_method calculation_method,
    p_qty NUMERIC,
    p_value NUMERIC,
    p_dimension_text TEXT,
    p_unit_price NUMERIC,
    p_dimensional_factor NUMERIC,
    p_imported_total_price NUMERIC
)
RETURNS NUMERIC AS $$
DECLARE
    v_result NUMERIC(20, 4);
    v_dimension_num NUMERIC;
BEGIN
    IF p_unit_price IS NULL THEN
        p_unit_price := 0;
    END IF;

    CASE p_calc_method
        WHEN 'VALUE_X_UNIT_PRICE' THEN
            v_result := COALESCE(p_value, 0) * p_unit_price;

        WHEN 'QTY_X_UNIT_PRICE' THEN
            v_result := COALESCE(p_qty, 0) * p_unit_price;

        WHEN 'FIXED_TOTAL' THEN
            v_result := COALESCE(p_imported_total_price, 0);

        WHEN 'DIMENSIONAL_WEIGHT' THEN
            -- For special rows like:
            -- dimension * qty * value * factor / 1,000,000
            -- Only works if dimension_text is numeric.
            BEGIN
                v_dimension_num := NULLIF(regexp_replace(COALESCE(p_dimension_text, ''), '[^0-9\.]', '', 'g'), '')::NUMERIC;
            EXCEPTION WHEN others THEN
                v_dimension_num := 0;
            END;

            v_result :=
                COALESCE(v_dimension_num, 0)
                * COALESCE(p_qty, 0)
                * COALESCE(p_value, 0)
                * COALESCE(p_dimensional_factor, 1)
                / 1000000;

        WHEN 'ZERO_OR_INFO' THEN
            v_result := 0;

        WHEN 'MANUAL' THEN
            v_result := COALESCE(p_imported_total_price, 0);

        ELSE
            -- UNKNOWN: safest fallback is imported total price
            v_result := COALESCE(p_imported_total_price, 0);
    END CASE;

    RETURN ROUND(COALESCE(v_result, 0), 4);
END;
$$ LANGUAGE plpgsql IMMUTABLE;

-- =========================================================
-- 27. FUNCTION: CALCULATE OFFER
-- This reproduces Excel logic:
-- Part Unit Price = SUM(MTO rows)
-- Line Part Total = Qty * Part Unit Price
-- =========================================================

CREATE OR REPLACE FUNCTION calculate_offer(p_offer_id BIGINT, p_user_id BIGINT DEFAULT NULL)
RETURNS BIGINT AS $$
DECLARE
    v_run_id BIGINT;
    v_run_number INTEGER;
    v_total NUMERIC(20, 4) := 0;
BEGIN
    SELECT COALESCE(MAX(run_number), 0) + 1
    INTO v_run_number
    FROM calculation_runs
    WHERE offer_id = p_offer_id;

    INSERT INTO calculation_runs (
        offer_id,
        run_number,
        status,
        calculated_by
    )
    VALUES (
        p_offer_id,
        v_run_number,
        'calculated',
        p_user_id
    )
    RETURNING id INTO v_run_id;

    -- Insert calculation details at MTO row level.
    INSERT INTO calculation_details (
        calculation_run_id,
        offer_id,
        offer_scope_id,
        offer_line_id,
        offer_line_part_id,
        scope_id,
        part_id,
        material_id,
        mto_row_id,
        line_part_qty,
        mto_qty,
        mto_value,
        unit_id,
        project_unit_price,
        calculated_row_total,
        calc_method,
        warning_message
    )
    SELECT
        v_run_id,
        o.id AS offer_id,
        os.id AS offer_scope_id,
        ol.id AS offer_line_id,
        olp.id AS offer_line_part_id,
        s.id AS scope_id,
        p.id AS part_id,
        mr.material_id,
        mr.id AS mto_row_id,
        olp.qty AS line_part_qty,
        mr.qty AS mto_qty,
        mr.value AS mto_value,
        mr.unit_id,
        COALESCE(pmp.unit_price, mr.imported_unit_price, 0) AS project_unit_price,

        calculate_mto_row_total(
            mr.calc_method,
            mr.qty,
            mr.value,
            mr.dimension_text,
            COALESCE(pmp.unit_price, mr.imported_unit_price, 0),
            mr.dimensional_factor,
            mr.imported_total_price
        ) * olp.qty AS calculated_row_total,

        mr.calc_method,

        CASE
            WHEN pmp.unit_price IS NULL AND mr.material_id IS NOT NULL
                THEN 'Project material price not found; imported price used.'
            WHEN mr.calc_method = 'UNKNOWN'
                THEN 'Unknown calculation method.'
            ELSE NULL
        END AS warning_message

    FROM offers o
    JOIN offer_scopes os ON os.offer_id = o.id
    JOIN scopes s ON s.id = os.scope_id
    JOIN offer_lines ol ON ol.offer_scope_id = os.id
    JOIN offer_line_parts olp ON olp.offer_line_id = ol.id
    JOIN parts p ON p.id = olp.part_id
    JOIN mto_rows mr
        ON mr.mto_version_id = o.mto_version_id
       AND mr.scope_id = s.id
       AND mr.part_id = p.id
       AND mr.is_active = TRUE
       AND mr.is_price_relevant = TRUE
    LEFT JOIN project_material_prices pmp
        ON pmp.offer_id = o.id
       AND pmp.material_id = mr.material_id
    WHERE o.id = p_offer_id;

    -- Update cached unit_price and total_price on offer_line_parts
    WITH part_totals AS (
        SELECT
            offer_line_part_id,
            SUM(calculated_row_total) AS total_price,
            MAX(line_part_qty) AS qty
        FROM calculation_details
        WHERE calculation_run_id = v_run_id
        GROUP BY offer_line_part_id
    )
    UPDATE offer_line_parts olp
    SET
        total_price = pt.total_price,
        unit_price = CASE
            WHEN pt.qty > 0 THEN ROUND(pt.total_price / pt.qty, 4)
            ELSE 0
        END,
        calculation_status = 'calculated',
        last_calculated_at = now(),
        updated_at = now()
    FROM part_totals pt
    WHERE pt.offer_line_part_id = olp.id;

    SELECT COALESCE(SUM(calculated_row_total), 0)
    INTO v_total
    FROM calculation_details
    WHERE calculation_run_id = v_run_id;

    UPDATE calculation_runs
    SET
        total_material_cost = v_total,
        total_offer_price = v_total,
        warnings_count = (
            SELECT COUNT(*)
            FROM calculation_details
            WHERE calculation_run_id = v_run_id
              AND warning_message IS NOT NULL
        ),
        errors_count = (
            SELECT COUNT(*)
            FROM calculation_details
            WHERE calculation_run_id = v_run_id
              AND error_message IS NOT NULL
        ),
        status = CASE
            WHEN (
                SELECT COUNT(*)
                FROM calculation_details
                WHERE calculation_run_id = v_run_id
                  AND error_message IS NOT NULL
            ) > 0 THEN 'has_error'::calculation_status

            WHEN (
                SELECT COUNT(*)
                FROM calculation_details
                WHERE calculation_run_id = v_run_id
                  AND warning_message IS NOT NULL
            ) > 0 THEN 'has_warning'::calculation_status

            ELSE 'calculated'::calculation_status
        END
    WHERE id = v_run_id;

    -- Create snapshot
    INSERT INTO offer_price_snapshots (
        offer_id,
        calculation_run_id,
        version_number,
        total_price,
        snapshot_json,
        created_by
    )
    VALUES (
        p_offer_id,
        v_run_id,
        v_run_number,
        v_total,
        jsonb_build_object(
            'offer_id', p_offer_id,
            'calculation_run_id', v_run_id,
            'total_price', v_total,
            'created_at', now()
        ),
        p_user_id
    );

    RETURN v_run_id;
END;
$$ LANGUAGE plpgsql;

-- =========================================================
-- 28. FUNCTION: COPY DEFAULT MATERIAL PRICES TO OFFER
-- Should be called after creating a new offer.
-- =========================================================

CREATE OR REPLACE FUNCTION initialize_offer_material_prices(p_offer_id BIGINT)
RETURNS INTEGER AS $$
DECLARE
    v_count INTEGER;
BEGIN
    INSERT INTO project_material_prices (
        offer_id,
        material_id,
        unit_id,
        dimension_text,
        default_unit_price,
        unit_price,
        price_source,
        is_overridden
    )
    SELECT
        p_offer_id,
        m.id,
        m.default_unit_id,
        m.default_dimension,
        COALESCE(m.default_unit_price, 0),
        COALESCE(m.default_unit_price, 0),
        'default',
        FALSE
    FROM materials m
    WHERE m.is_active = TRUE
    ON CONFLICT (offer_id, material_id) DO NOTHING;

    GET DIAGNOSTICS v_count = ROW_COUNT;
    RETURN v_count;
END;
$$ LANGUAGE plpgsql;

-- =========================================================
-- 29. TRIGGER: MARK OFFER LINE PARTS OUTDATED WHEN PROJECT MATERIAL PRICE CHANGES
-- =========================================================

CREATE OR REPLACE FUNCTION mark_offer_calculations_outdated()
RETURNS TRIGGER AS $$
BEGIN
    UPDATE offer_line_parts olp
    SET
        calculation_status = 'outdated',
        updated_at = now()
    FROM offer_lines ol
    JOIN offer_scopes os ON os.id = ol.offer_scope_id
    WHERE olp.offer_line_id = ol.id
      AND os.offer_id = NEW.offer_id;

    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_project_material_prices_mark_outdated
AFTER UPDATE OF unit_price ON project_material_prices
FOR EACH ROW
WHEN (OLD.unit_price IS DISTINCT FROM NEW.unit_price)
EXECUTE FUNCTION mark_offer_calculations_outdated();

-- =========================================================
-- 30. TRIGGER: MARK LINE PART OUTDATED WHEN QTY OR PART CHANGES
-- =========================================================

CREATE OR REPLACE FUNCTION mark_line_part_outdated()
RETURNS TRIGGER AS $$
BEGIN
    NEW.calculation_status = 'outdated';
    NEW.last_calculated_at = NULL;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_offer_line_parts_mark_outdated
BEFORE UPDATE OF qty, part_id ON offer_line_parts
FOR EACH ROW
WHEN (
    OLD.qty IS DISTINCT FROM NEW.qty
    OR OLD.part_id IS DISTINCT FROM NEW.part_id
)
EXECUTE FUNCTION mark_line_part_outdated();

6. چرا calculation_method مهم است؟
در اکسل مشخص شد که همه ردیف‌ها یک مدل محاسبه ندارند.

پس این enum گذاشته شده است:

CREATE TYPE calculation_method AS ENUM (
    'VALUE_X_UNIT_PRICE',
    'QTY_X_UNIT_PRICE',
    'DIMENSIONAL_WEIGHT',
    'FIXED_TOTAL',
    'ZERO_OR_INFO',
    'MANUAL',
    'UNKNOWN'
);
معنی هرکدام:

مقدار	کاربرد
VALUE_X_UNIT_PRICE	حالت اصلی: VALUE × UNIT_PRICE
QTY_X_UNIT_PRICE	وقتی VALUE خالی است و محاسبه با Qty انجام می‌شود
DIMENSIONAL_WEIGHT	برای مواردی مثل 7.86 و محاسبه ابعادی
FIXED_TOTAL	وقتی مقدار Total از اکسل ثابت وارد می‌شود
ZERO_OR_INFO	ردیف‌های اطلاعاتی یا بدون قیمت
MANUAL	مواردی که دستی تعیین می‌شوند
UNKNOWN	موارد مشکوک که باید بررسی شوند

خطاهای داده‌ای مثل قیمت خالی، متریال تکراری و محاسبه خاص قابل کنترل هستند.

برای آینده، سطح دسترسی، خروجی PDF/Excel، ضرایب سود و مالیات قابل اضافه شدن است.

21. User Flow پیشنهادی با جزئیات
Flow 1: ساخت یک آفر کامل
1. User logs in
2. User opens Offer List
3. User clicks New Offer
4. User fills:
   - Project Name
   - Offer Number
   - Type
   - Input Date
   - Close Date
5. System creates Offer
6. System copies default material prices to project
7. User goes to Offer Detail
8. User opens Scopes & Lines
9. User clicks Add Scope
10. User selects 1000A AL
11. User clicks Add Line
12. User enters Line1
13. User clicks Add Part
14. User selects FEEDER-1000 and Qty 1
15. User selects Flexible connection and Qty 1
16. User selects TOP OFF BOX 250 and Qty 2
17. User adds more scopes/lines
18. User opens Material Prices
19. User adjusts project-specific material prices if needed
20. User clicks Calculate
21. System calculates all prices
22. User reviews Calculation tab
23. User opens Reports
24. User exports or views customer summary
25. User changes status to Ready or Sent
Flow 2: تغییر قیمت متریال پروژه
1. User opens Offer
2. User goes to Material Prices
3. User searches material
4. User changes Project Price
5. System marks offer calculation as Outdated
6. Header shows: Needs Recalculation
7. User clicks Calculate
8. New calculation run is saved
9. New total price appears
10. History logs price change and calculation
Flow 3: اضافه کردن Line جدید بعد از محاسبه
1. User opens Offer
2. Current status is Calculated
3. User opens Scopes & Lines
4. User adds new Line
5. User adds Parts and Qty
6. System marks calculation as Outdated
7. User clicks Recalculate
8. System creates new calculation run
9. Report updates
Flow 4: بررسی جزئیات قیمت یک قطعه
1. User opens Calculation tab
2. User clicks on Scope 1000A AL
3. User clicks Line1
4. User clicks FEEDER-1000
5. System opens Part Calculation Detail
6. User sees all MTO rows/materials used in this Part
7. User checks Material, Value, Unit Price, Row Total

یعنی کاربر اول آفر را می‌سازد، بعد اسکوپ و لاین و قطعات را تعریف می‌کند، سپس قیمت متریال را در صورت نیاز اصلاح می‌کند، بعد محاسبه می‌گیرد.

اما از نظر UX بهتر است در صفحه Offer Detail یک Progress کوچک هم داشته باشیم:

1. اطلاعات پروژه
2. قیمت متریال
3. اسکوپ و لاین
4. محاسبه
5. خروجی
و وضعیت هر مرحله مشخص باشد:

Complete
Needs Attention
Not Started
این باعث می‌شود کاربر گم نشود و بداند برای تکمیل آفر چه چیزی کم است.
